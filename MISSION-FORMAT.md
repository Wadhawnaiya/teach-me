# MISSION.md Format

`MISSION.md` lives at the workspace root. It is the single source of truth for *why* the user is investing time. Every teaching choice — scope, depth, examples, pace, resources — must be traceable to it.

## Template

```md
# Mission: {Topic}

## Why
{1-3 sentences. The concrete, real-world outcome the user is chasing. What will actually change in their life, career, identity, or contribution when they have this? Avoid vague "understand X".}

## Success looks like
- {Specific, observable capability or result the user will be able to demonstrate}
- {Another measurable thing}
- {…}

## Constraints
- {Time available per week, deadlines, budget, device/data limits, language preference, prior commitments, health, etc.}

## Out of scope (for now)
- {Adjacent topics explicitly deprioritized to protect focus and ZPD}
```

## Rules

- **One mission per workspace**. Multiple unrelated goals = multiple workspaces.
- **Concrete & observable**. "Ship a production feature that reduces support tickets by 30%" beats "get better at backend".
- **Push for the real why**. Interview until the mission feels alive and motivating.
- **Revise with evidence**. When learning records show the user's actual goal has evolved, update the mission (with their explicit agreement) and record the shift.
- **Keep it short**. A mission longer than a screen has stopped being a compass.
- **Revisit regularly**. Start sessions by checking alignment with the mission.

Write the first version only after a short, high-quality conversation that surfaces real motivation.
